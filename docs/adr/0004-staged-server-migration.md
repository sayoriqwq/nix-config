# ADR-0004：服务器采用分阶段迁移

- 状态：Accepted
- 日期：2026-07-17

## 背景

当前服务器运行 Ubuntu，目标是迁移到 NixOS。服务器同时承载可变业务数据、服务配置、网络入口和远程管理能力。直接在一个变更中完成系统重装、存储重构、服务声明化和数据迁移，会把多个故障域叠加在一起，并可能导致无法启动、SSH 失联或数据不可恢复。

## 决策

服务器迁移拆成彼此可验证的阶段：

1. **Ubuntu 过渡期**：只使用 standalone Home Manager 管理最小用户环境，不接管 boot、apt、systemd 系统服务、网络、防火墙或业务 daemon。
2. **最小 NixOS 设计**：根据真实 inventory 编写最小 NixOS output 与 disko 布局，只包含启动、磁盘、网络、SSH、管理用户、sudo、基础防火墙和 provider 必需设置。
3. **安装前验证**：先 build，再运行 nixos-anywhere `--vm-test`，验证分区、挂载、启动与 SSH；生产服务器不在此阶段重装。
4. **人工批准迁移**：完整备份、恢复验证、provider console/rescue、目标磁盘、boot mode、网络和 SSH 恢复路径全部确认后，才允许在人工监督下执行正式安装。
5. **业务恢复**：NixOS 最小系统稳定后，优先原样恢复现有 Compose/容器与数据；系统迁移成功和业务恢复成功分别验收。
6. **后续声明化**：只有业务恢复稳定后，才为适合的服务逐项建立独立 Issue，迁移到 NixOS module 或 `virtualisation.oci-containers`。

首轮服务器迁移不同时引入 ZFS、LUKS、impermanence、复杂 LVM、数据库大版本升级或业务栈重写。

## 人工关卡

以下动作必须获得当前 Issue 中明确、当次的维护者批准，并由维护者执行或实时监督：

- 修改远程网络、DNS、防火墙或 SSH 访问；
- 运行 disko、nixos-anywhere、格式化或重分区；
- 重启、关机或替换服务器；
- 恢复、迁移或删除生产数据；
- 将 Draft PR 标记为可合并并合并。

Agent 默认停在“配置、检查、命令和回滚清单已准备完成”处。

## 结果

### 正面结果

- 系统安装、网络恢复和业务恢复拥有独立的故障边界；
- 每个阶段可以单独 build、测试、回滚和审计；
- 保留现有 Compose 可减少重装当天的变量；
- 生产操作具有明确的人类责任和批准记录。

### 代价

- 迁移需要更多阶段和文档；
- Ubuntu 过渡配置与最终 NixOS 配置会短期并存；
- 部分服务不会立即变成“纯 Nix”管理。

这些代价低于一次性迁移造成不可恢复中断的风险。

## 被否决的方案

### 一次 PR 完成全部迁移

否决原因：同时改变 OS、磁盘、网络、服务定义和数据格式，无法可靠定位故障，也缺少可控回滚点。

### 先重写全部 Compose 为 NixOS 服务再重装

否决原因：把业务架构重构变成操作系统迁移的前置条件，扩大停机与验证范围。

### Agent 无人值守执行远程重装

否决原因：远程磁盘、网络、provider 行为和业务数据具有不可逆风险，必须保留实时人工关卡。
